3. Problemas reales detectados

Ahora lo importante: los problemas reales.

No son muchos, pero algunos sí importan.

PROBLEMA 1 — Race condition en reservas

Este bloque:

if (!existing) {
   await createReserva(...)
}

No está protegido por transacción.

Si dos usuarios reservan simultáneamente:

User A → confirm
User B → confirm

ambos pueden crear reserva.

Solución:

usar transacción serializable.

Ejemplo:

await prisma.$transaction(async (tx) => {

 const existing = await tx.reserva.findFirst(...)

 if (!existing) {
    await tx.reserva.create(...)
 }

})

Esto es muy importante para producción.

PROBLEMA 2 — deduplicación de mensajes WA

No aparece en este archivo.

Si WhatsApp reintenta el webhook (muy común):

podrías tener:

mensaje duplicado
bot responde duplicado

Esto debería resolverse en:

webhookProcessor

con:

waMessageId UNIQUE
PROBLEMA 3 — hardcode de departamentos

Esto:

const ALL_HABITATIONS = [
 'Pewmafe',
 'Luminar Mono',
 'Luminar 2Amb',
 'LG'
]

es frágil.

Si agregás uno nuevo:

rompe la lógica

Mejor:

const complejos = await prisma.complejo.findMany()
PROBLEMA 4 — lógica de fotos puede dispararse mal

Este regex:

/fotos?|imagene?s?|pictures?|mostrame|ver el depto/i

puede dispararse con frases como:

quiero ver el depto disponible

y enviar fotos inesperadamente.

Mejor:

detectar también intención:

intent === consulta_fotos
PROBLEMA 5 — falta timeout de Claude

Estas llamadas no tienen timeout:

classifyIntent
generateResponse

Si Claude tarda:

WhatsApp queda colgado

Recomendado:

timeout 3–5s
fallback response
PROBLEMA 6 — acumulación de entidades depende del bot

Tu memoria se basa en:

metadata.entities
de mensajes del BOT

Si Claude falla en una respuesta:

entidad perdida

Es mejor guardar entidades también cuando las dice el usuario.

PROBLEMA 7 — costo potencial del LLM

Cada mensaje ejecuta:

classifyIntent
generateResponse

Sin rate limit.

Un usuario puede enviar:

20 mensajes seguidos

y disparar muchas llamadas.

Recomendado:

rate limit por conversacion
